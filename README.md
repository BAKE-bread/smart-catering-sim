# 智能餐饮匹配协议并发仿真系统 (‘ICMP’-Sim)
## Intelligent Catering Matching Protocol Concurrent Simulation System

## 📌 项目简介

本项目是一个基于 Python 多线程的分布式并发仿真系统，模拟了“多厨师组、多顾客”的智能点餐匹配场景。系统核心实现了一套基于特征二分查找的降阶搜索协议，并引入了带有 LRU 缓存的“厨师学习机制”与“全局知识共享池”。

通过模拟顾客与厨师组之间的异步通信，系统展示了在高并发状态下，局部缓存与全局状态同步如何有效减少匹配交互轮次，并通过探针（Probe）机制实现点餐过程的加速。近期版本引入了噪音与容错测试，更拟真地模拟了现实环境中的信息交互偏差。

## ✨ 核心特性

- **高并发交互模型**：使用原生 `threading` 与线程安全队列 `queue.Queue` 模拟异步点餐与响应。
- **智能特征降阶匹配**：顾客通过动态主材与辅料特征发起排他性提问，模拟二分查找，将匹配复杂度从 **O(N)** 降至 **O(log N)**。
- **探针加速与自适应学习**：厨师具备基于 LRU 策略的本地知识库，可从成功订单中学习映射关系；支持带延迟模拟的全局知识共享。
- **灵活网络拓扑**：支持自定义厨师组与顾客组的交叉映射关系，按幂律分布模拟真实世界的不均匀调度。
- **抗干扰噪音测试**：支持注入“顾客记错配料”或“厨师信息同步延迟”等模拟噪音，检验协议的鲁棒性。

## 🛠️ 运行环境与依赖

- **运行环境**：Python 3
- **核心依赖**：`threading`, `queue`, `dataclasses`, `typing` (均为 Python 标准库)
- **分析依赖**：`matplotlib`, `numpy` (用于运行结束后的数据分析与多维度图表生成)

## ⚙️ 全局配置与参数修改入口

系统的主要控制枢纽为 `config.py` 文件。用户所有的参数调优、机制开关与实验配置都应在此文件中进行修改。

### 1. 核心规模与仿真控制

| 常量名 | 类型 | 建议范围 | 说明 |
|--------|------|----------|------|
| `N` | `int` | 1 ~ 2000 | 核心规模：总菜谱数/菜单数。决定了特征空间维度。 |
| `SIMULATION_SPEED` | `float` | 0.001 ~ 0.1 | 终端打印延迟（秒），控制输出流速度以便观察。 |
| `CUSTOMER_ARRIVAL_*` | `float` | ≥ 0 | 顾客到达餐厅的最小/最大间隔时间（秒）。 |

### 2. 学习机制与知识库配置

| 常量名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `ENABLE_CHEF_LEARNING` | `bool` | `True` | 是否允许厨师记录成功的匹配映射。 |
| `ENABLE_SHARED_KNOWLEDGE` | `bool` | `True` | 是否开启全局知识同步机制。 |
| `LOCAL_KNOWLEDGE_CAPACITY` | `int` | 100 | 单个厨师本地 LRU 缓存的容量上限。 |
| `KNOWLEDGE_SYNC_DELAY_PROB` | `float` | 0.1 | 全局知识同步时发生延迟的概率。 |

### 3. ⚠️ 噪音与容错测试 (Noise Options)

为了测试绝对二分查找在现实中的脆弱性，可在 `config.py` 开启以下噪音：

- `ENABLE_CUSTOMER_NOISE` (默认 `False`)：模拟顾客记忆错乱。顾客有 `CUSTOMER_NOISE_PROB` 的概率将“包含”某食材记反为“不包含”。
- `ENABLE_CHEF_NOISE` (默认 `False`)：模拟厨师因知识库同步延迟而导致的误答（状态更新与口头回答解耦防崩溃）。

**测试效果**：开启噪音会导致二分查找候选集被错误排除，系统匹配准确率将大幅下跌，原本 100% 正确的匹配结果可能会产生错误学习，进而污染全局知识库。

## 🌐 网络拓扑与资源分配

系统的网络拓扑结构在 `config.py` 的 `CHEF_CUSTOMER_GROUPS` 中定义。

- **厨师 ID**：可重复出现在多个组，实现一人多职的交叉分配。
- **顾客 ID**：不可重复。每个顾客在逻辑上只能存在于一个顾客组中，否则可能导致状态机错乱。
- **生成器**：默认通过 `generate_chef_customer_groups` 根据幂律分布（Zipf）自动生成符合真实情况的不均匀群组。

## 🧠 特征不变量与数学解析

在未开启噪音干扰的理想运行态中，可观察到以下由数学逻辑主导的“不变量”：

### 1. 探针匹配平均轮次恒为 1

**原因**：协议设计顾客开口的第一句话（第 0 轮）永远是直接报菜谱 ID 的探针。只要厨师（通过本地或共享）认识这个 ID，一问一答即刻结束。中间没有过渡，物理交互轮次永远被截断为严格的 1 轮。

### 2. 学习提升效率固化 (当 N=200 时在 87% 附近)

**原因**：系统学习提升率公式为  
\[
\text{提升率} = \frac{\text{二分查找期望轮次} - 1}{\text{二分查找期望轮次}} \times 100\%
\]

在生成逻辑中，特征空间由二进制位构成，主材数量精确等于 \(\lceil \log_2 N \rceil\)。  
当 \(N = 200\) 时，理想的二分查找期望轮次接近 \(\log_2 200 \approx 7.64\) 轮。  

代入公式：  
\[
\frac{7.64 - 1}{7.64} \approx 0.869 \quad \Rightarrow \quad 86.9\%
\]

这一数字是系统采用的对数函数与探针截断机制共同铸就的常数。

## ⚠️ 边界情况与已知限制

- **非法 N 值**：`N <= 0` 会引发底层数学库对数运算崩溃。
- **死锁等待**：若顾客组被分配了一个未实例化的“幽灵厨师 ID”，该组顾客请求将永远挂起。
- **全局锁瓶颈**：由于 Python 的 GIL（全局解释器锁）以及保证状态一致性所使用的 `simulation_data_lock` 等强锁，当并发数达到数千甚至过万级别时，系统会因为锁争夺和上下文切换导致性能严重下降。
- **二分查找的脆弱性**：若开启配置中的“顾客噪音”，由于当前的特征排除逻辑缺乏回退机制，可能会导致候选菜单直接清零（Empty Candidates），因此已在 `chef.py` 补充了防崩溃校验，允许带着错误的信息完成匹配（从而记录错误率）。
